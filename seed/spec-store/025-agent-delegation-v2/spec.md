# Feature Specification: Agent Delegation v2 (unified registry, ephemeral agents, surface agents)

**Feature Branch**: `025-agent-delegation-v2`

**Created**: 2026-07-13

**Status**: Draft — revised after independent review (see `spec-review.md`); all findings resolved below.

**Relationship to `016-unified-agents`**: This is a **delta spec**, not a replacement. `016` is left untouched. `016` described the *aspiration* ("one Agent concept, one allowlist governs it everywhere") but was written against the CopilotKit-era client, before the v2 server-owned-run rewrite — its gating mechanism (`CopilotProvider`'s per-action `available` flag) no longer exists. In practice the aspiration was never fully realized: a second, legacy tool-resolution engine (`src/lib/agent/subagents/tools.ts`) still exists purely for delegated execution, completely disjoint from the v2 registry (`src/lib/assistant/registry.ts`) that governs everything else. This spec finishes what `016` started, on top of the current (v2) architecture, and adds two new capabilities (ephemeral agents, surface agents) that fell out of closing that gap. **Once implemented and tested, the intent is to fold this spec's content into `016` and retire this file** — it is scaffolding for that merge, not a permanent parallel spec.

**Input**: Synthesized from a live debugging/design session. A real user trajectory showed the default "assistant" agent fail to use the UI Preview app at all: `bos_app_list` doesn't list it (deliberately hidden), and `find_tools` came back empty for every query — not because of bad search terms, but because the tool ids were entirely absent from that agent's allowlist, and `find_tools` structurally cannot surface a tool that isn't already in `gate.allow`. Investigating delegation specifically (would `agent_delegate` to a differently-configured agent have helped?) surfaced a deeper problem: delegated execution runs through an entirely separate, legacy tool catalog (`subagents/tools.ts`) that predates the v2 rewrite and was never updated with newer tools (UI Preview, A2UI, and anything else added directly to the v2 registry since) — so even an agent whose `AGENT.md` lists these tools has them **silently dropped** the moment it's reached via delegation rather than as the active/primary agent. Research into how Claude Code and OpenCode structure delegation confirmed the fix: both use exactly one tool catalog; "being delegated to" only ever changes which subset of that one catalog is visible, never which catalog resolves it. Working through the design surfaced two further wanted capabilities (ephemeral, unpersisted one-off agents; and app-scoped "surface agents" that exist only while their window is open) and a transcript/continuity design (delegation collapses to one tool-call+result in the parent's history; delegated contexts never retain memory across separate calls). A subsequent independent review (by an agent with no knowledge of the design conversation, spot-checked against the codebase) found the core design sound but under-specified in several places — those findings are resolved throughout this revision, primarily in the second Clarifications session and FR-015 onward. (Note for future readers: an unrelated, already-shipped feature — deferred-tool discovery on branch `bos/deferred-tools-discovery` — happened to reuse "025" informally in commit messages before this spec claimed the number in the spec-store; there is no actual numbering collision, since no `025-deferred-tool-discovery/spec.md` was ever committed, but the two are easy to conflate when grepping history.)

## Clarifications

### Session 2026-07-13

- Q: Should BOS keep two tool-resolution engines — the v2 registry for the active/primary agent, and the legacy `subagents/tools.ts` catalog for delegated execution? → A: **No.** Retire the legacy engine entirely. One registry (v2, `assistantTools()`/`gate.ts`) resolves tools in every execution context — active personality and every kind of delegation alike. Confirmed against real-world precedent: neither Claude Code nor OpenCode has two parallel tool-resolution engines; delegation only ever changes which subset of one catalog a given execution can see, never which catalog.
- Q: How should a completely ad-hoc, one-off, unpersisted agent be defined? → A: An **ephemeral agent** — described purely by `{name, systemPrompt, type}` at the moment of delegation, with no `AGENT.md` ever written to disk. Mirrors Claude Code's `general-purpose` Task pattern (no config file needed for the ad-hoc case; a config file is only for agents meant to be reusable/named).
- Q: What tools does an ephemeral agent get? → A: **All of the delegating agent's currently-allowed tools, filtered to server-executable ones only** (`execution: "server"`) — see FR-003 and A1 (Session 2) for the precise deferred/revealed semantics. An ephemeral agent spec has no `tools:` field at all — its set is always *derived*, never separately configured, so there is no code path where it can end up with an unset/empty tool list by omission. Frontend/Tier-2 tools are deliberately excluded from this inheritance — not because an inner loop lacks the attachment to run them (it shares the parent run's `awaitFrontendResult`, same as a surface agent's own tools, per FR-011), but as a scope line: open-ended "inherit everything" is kept to safe, stateless server capabilities, while any deliberate reach into browser/window UI goes through a **surface agent**'s bounded, app-declared `toolNames` instead. A named agent delegated to has no such restriction — its own configured `tools:` may include frontend tools, and they dispatch correctly for the same reason.
- Q: Should an ephemeral agent's inherited tools be deferred (hidden until revealed via `find_tools`)? → A: **No** — full visibility immediately. It exists for one focused task, described in full at creation time; a discovery round-trip adds latency for no benefit.
- Q: What changes for a persisted, named agent (has an `AGENT.md`)? → A: Nothing about *what* it configures — it keeps its own explicit `tools:`/`deferredTools:` allowlist exactly as today. What changes is *how* that allowlist is resolved: always through the single v2 registry, in every context, never through the legacy catalog.
- Q: Can an app expose a specialized agent — not just tools — that only exists while its window is open? → A: **Yes** — a **surface agent**: `{name, systemPrompt, toolNames}`, registered by a mounted app window the same way surface tools already are (`registerAppSurfaceTools`'s sibling), discoverable via `agent_list`/`find_agent` only while that window stays registered, and gone the instant it unmounts.
- Q: Should delegating to any of these (named, ephemeral, or surface) spin up a genuinely separate run/conversation for full context isolation? → A: **No** — and not raw inline steps of the parent's own loop either. Delegation executes as an **inner loop**: a nested tool-loop sharing the delegating run's own dispatch machinery (`emit`, `awaitFrontendResult`), so any inherited Tier-2 tools (a surface agent's own app tools) dispatch to the exact browser tab already attached to that run. No new run, no cross-run event relay, no separate conversation record is ever created.
- Q: Does the inner loop see the parent conversation's history? → A: **No.** Blank slate, seeded only by the `task` string given at the point of delegation — the same convention already used by every existing BOS delegate (Nora/Clark/Elon/Dave): the sub-agent does not see the conversation; be specific in the instructions.
- Q: Do the inner loop's own intermediate steps get persisted as ordinary conversation messages? → A: **No.** They are reported as nested progress events (extending the existing `ctx.onEvent` / `encodeNested()` pattern `agent_delegate` already uses for its legacy sub-agent calls), never written via `io.saveMessages()`. The delegating run's persisted transcript gains exactly **one** `tool_call` + **one** `tool_result` for the whole delegation, regardless of how many inner steps it took.
- Q: Does a delegation target retain memory/state across *separate* delegation calls (e.g. two calls to the same surface agent, later in the same conversation)? → A: **No, never — zero exceptions.** Each call is fully blank-slate. Continuity across related delegations ("build a mockup," then later "make the buttons bigger") is entirely the *delegating* agent's responsibility: every past delegation is already recorded in its own transcript as one clean tool-call+result pair, so it always has what it needs to compose a sufficiently detailed task string next time, without the delegate needing — or being permitted — any memory of its own.
- Q: What happens to a tool id that fails to resolve (typo, removed tool, an inherited/derived set referencing something gone)? → A: It **must be surfaced**, never silently dropped, per the standing no-silent-failures policy. See FR-023.
- Q: Is nested delegation (a delegate itself calling `agent_delegate`) still possible, and still guarded? → A: Yes in principle, but see Session 2 (A4) — today's guard is narrower than first assumed and this spec must actually generalize it, not merely preserve it.

### Session 2026-07-13 (post independent review)

An independent review (`spec-review.md`, verified against the codebase, no knowledge of the design conversation above) found the core design sound but flagged eight gaps and four ambiguities. Resolved here; each new/changed FR is cited.

- Q: The spec's Key Entities said the `Agent` shape is "unchanged" but omitted `model`, and never addressed which model an inner loop uses. → A: A named agent's `model` override, if set, applies to its inner loop; ephemeral and surface agents have no `model` field and always run on the delegating run's currently configured model. **FR-015.**
- Q: What about a named agent's `skills`/`mcp` allowlists and `useDefaultPrompt` flag under delegation? → A: `skills`/`mcp` resolve through the unified registry exactly like `tools` (no behavior change, just one less code path). An ephemeral agent additionally inherits the delegating agent's `skills` and `mcp` allowlists in full, for the same reason it inherits server tools. `useDefaultPrompt` does **not** apply to ephemeral or surface agents — their system prompt is used verbatim; only named agents retain today's opt-in default-prompt-prepending behavior. **FR-016, FR-017.**
- Q: What step cap, timeout, and exhaustion behavior does the inner loop get? Today's numbers already disagree (legacy `runToolLoop` default 8, dev/spec-delegate variant 40, v2 main loop 24). → A: The inner loop gets its own explicit, finite cap (exact default decided in `plan.md`, informed by — not bound to — today's numbers; a dev/spec-heavy delegation MAY warrant a larger cap than a quick ephemeral one). Inner-loop steps never count against the delegating run's own budget. On exhaustion, the inner loop returns an in-band partial-completion summary (mirroring the existing step-limit message shape), never a silent stop. **FR-018.**
- Q: What exactly is returned as the delegation's `tool_result`, and what happens on failure? → A: The delegate's own final natural-language answer, used verbatim as the `tool_result` (no separate truncation/summarization step is introduced by this mechanism). Any inner-loop failure — model/provider error, cap exhaustion, target-agent-not-found, depth-guard triggered — MUST reach the delegating agent as an in-band error string, never an unhandled exception or a silent no-op. `contentOnly` (today's existing `agent_delegate` input) is untouched by this spec: it remains meaningful only for `type: "claude"` delegation (feature-branch gating), which this spec does not change. **FR-019, FR-020.**
- Q: How are surface-agent ids derived, and what happens on collision with a persisted agent, or with another surface agent (e.g. two windows of the same app)? → A: Slugify `name` (same function already used for ephemeral ids). Colliding with an existing **persisted** agent's id is a registration error — a surface agent may never shadow a persisted one, and this is surfaced at registration time, not silently resolved. Colliding with **another currently-registered surface agent** (the two-windows case) is auto-disambiguated with a short suffix derived from `windowId`. `agent_delegate`'s resolution checks the persisted roster before the surface-agent registry. **FR-022.**
- Q: `find_agent` matches by query against name/description — but the Surface Agent shape has no `description`. → A: Add one; apps SHOULD supply a real one-line description (matching), not rely on `name`/`systemPrompt` matching alone. **FR-008.**
- Q: Can two delegations (from the same turn, or a nested delegate and a sibling) ever run concurrently within one run, and does that need a serialization or event-correlation mechanism? → A: They cannot, by construction: the outer loop (and, transitively, every inner loop) executes its own tool calls strictly sequentially — one call, including any inner loop it spawns, fully resolves before the next begins. No new mechanism is needed; this is a property of the existing execution model, stated explicitly so implementers don't build one. **FR-021.**
- Q: Ephemeral agents can specify `type: "claude"` (and an existing `ephemeralSubagentType` field) — does FR-002/FR-003's inheritance model apply to them? → A: No. FR-002/003/004/016/017 (definition, tool inheritance/visibility, skill/mcp inheritance, default-prompt exclusion) apply to `type: "local"` only. Ephemeral `type: "claude"` is unaffected by this spec's registry unification and continues exactly as today, routed to `runClaudeAgent`, which provisions its own tools via the harness independently of the v2 registry. (FR-018's step cap is NOT in this "local only" list — it governs every inner loop uniformly; `type: "claude"` delegation simply never enters the inner loop at all, so the question doesn't arise for it.) **FR-002 (revised), FR-024(a).**
- Q: Does a window granting its surface agent access to arbitrary registered server tools (or its own Tier-2 tools) need any additional restriction beyond today's implicit trust model? → A: This is the intended mechanism, not a bypass to police — it's precisely how the motivating bug gets fixed (a persona lacking a tool can delegate to one that has it). No additional restriction for **built-in** apps, which are already as trusted as the rest of BOS's own code. Extending surface-agent registration to sandboxed/installed (less-trusted) apps, if that's ever supported at all, needs its own dedicated trust-boundary review — out of scope here, and not assumed. **FR-025.**

## User Scenarios & Testing *(mandatory)*

### User Story 1 - One catalog, no more silent gaps (Priority: P1)

Whichever way an agent is invoked — as the active/primary personality, or delegated to (named, ephemeral, or surface) — it resolves its tools from the exact same registry. No tool that exists in BOS is ever unreachable, with no explanation, purely because of *how* an agent happens to be invoked.

**Why this priority**: This is the actual bug that started the investigation, and the structural root cause of an entire class of confusing failures.

**Independent Test**: Delegate to an agent whose `AGENT.md` lists a v2-only tool (e.g. `build-studio`'s `ui_preview_render`) and confirm it's actually callable in the delegated run, not silently absent.

**Acceptance Scenarios**:

1. **Given** the `build-studio` agent is delegated to by another agent, **When** it runs, **Then** `ui_preview_render`/`a2ui_render` are present in its resolved tool set (previously: silently dropped, since the legacy catalog never defined them).
2. **Given** an agent's `tools:` allowlist contains an id that does not exist in the v2 registry, **When** that agent is loaded (as primary, or resolved for delegation), **Then** the gap is surfaced (logged and visible in Settings), not silently absent from its runnable set.

---

### User Story 2 - Ephemeral agents for one-off delegation (Priority: P1)

An agent can spin up a nameless, fileless, single-use specialist persona for a focused task without pre-authoring an `AGENT.md`, and that specialist actually has the tools, skills, and MCP access it needs.

**Why this priority**: Directly closes a second real bug found during design (an ephemeral "local" delegate today gets zero tools, since its constructed spec never sets a `tools:` field and empty is interpreted as "none").

**Independent Test**: Call `agent_delegate` with only `ephemeralName`/`ephemeralSystemPrompt`/`ephemeralType: "local"` (no existing `agent` id) and confirm the resulting persona can actually use tools.

**Acceptance Scenarios**:

1. **Given** the assistant delegates with only ephemeral fields, **When** the ephemeral agent runs, **Then** it has every server-executable tool the assistant currently has allowed, available immediately (no `find_tools` round-trip needed), plus the assistant's `skills`/`mcp` allowlists.
2. **Given** the same delegation, **When** the ephemeral agent's resolved tool set is inspected, **Then** it contains no frontend/Tier-2 tools, even if the assistant itself has some visible.
3. **Given** the assistant runs on a specific configured model, **When** an ephemeral agent is delegated to, **Then** it runs on that same model (it has no `model` field of its own to override it).

---

### User Story 3 - Delegation never pollutes the transcript (Priority: P1)

No matter how many steps a delegated task takes internally, the calling conversation's persisted history only ever grows by one tool-call and one tool-result. The internal steps are visible live as nested, expandable progress — never as separate top-level messages.

**Why this priority**: Without this, in-run persona switching (delegation sharing the parent run rather than spinning up an isolated one — see Assumptions for why full isolation was rejected) just relocates the noise problem true isolated delegation was invented to avoid.

**Independent Test**: Delegate a task that internally takes 5+ tool-call steps; inspect the saved conversation transcript afterward.

**Acceptance Scenarios**:

1. **Given** a delegation that takes 5 internal steps, **When** it completes, **Then** the outer conversation's saved transcript contains exactly one new `tool_call` and one `tool_result` (not five extra top-level messages).
2. **Given** the same delegation, **When** the user is watching live, **Then** the 5 internal steps are visible as a nested/expandable progress tree (matching today's `agent_delegate` card rendering), not as separate chat bubbles.
3. **Given** the inner loop instead exhausts its step cap without finishing, **When** it returns, **Then** the parent receives a clear in-band partial-completion summary as the `tool_result` (never a hang, exception, or silently-truncated answer).

---

### User Story 4 - Window-scoped surface agents (Priority: P2)

Opening an app (e.g. UI Preview) can make a specialized delegate agent available for exactly as long as that window stays open, with access to that app's own live frontend tools.

**Why this priority**: The motivating use case (a "Generative UI Agent" for UI Preview) — but P2 because it depends on User Stories 1–3 being correct first.

**Independent Test**: Open UI Preview, delegate a design task to its registered surface agent, confirm the mockup updates live; close the window; confirm the agent is no longer discoverable.

**Acceptance Scenarios**:

1. **Given** the UI Preview window is open, **When** `find_agent`/`agent_list` is called, **Then** its registered surface agent (e.g. "Generative UI Agent") is discoverable, with a real `description` and flagged as window-scoped.
2. **Given** a delegation to that surface agent, **When** it calls `a2ui_render` + `ui_preview_render` internally, **Then** the mockup visibly updates in the same, already-open window.
3. **Given** the UI Preview window is then closed, **When** `find_agent`/`agent_list` is called again, **Then** that surface agent is no longer discoverable, and delegating to it by name fails with a clear error (not a hang or a silent no-op).
4. **Given** two windows of the UI Preview app are somehow open simultaneously, **When** both register a surface agent with the same configured `name`, **Then** their derived ids are automatically disambiguated and both remain independently delegatable.

---

### User Story 5 - No delegation target ever remembers a prior call (Priority: P2)

Two separate delegations to the same target (named, ephemeral, or surface) never share state. Continuity, when needed, comes entirely from the delegating agent's own transcript and task descriptions.

**Why this priority**: Prevents a whole class of "why did it do that, I never told it that" bugs and dual-source-of-truth drift; P2 because it's a property to verify, not new surface area to build.

**Independent Test**: Delegate to the same surface agent twice, with an unrelated user/assistant exchange in between; confirm the second call has zero awareness of the first beyond what the task string explicitly says.

**Acceptance Scenarios**:

1. **Given** two sequential delegations to the same surface agent with an unrelated exchange in between, **When** the second delegation runs, **Then** it has no knowledge of the first delegation's internal steps or outcome except what the delegating agent explicitly includes in the second task string.

### Edge Cases

- An ephemeral agent's inherited tool set includes `agent_delegate` itself → the (now-generalized, see FR-024(b)) depth guard MUST still apply, preventing runaway nested ephemeral delegation.
- A surface agent's window re-registers mid-delegation (e.g. React re-render/hot-reload) → the in-flight inner loop MUST NOT be orphaned or duplicated.
- A surface agent's tool subset includes a Tier-2 tool the window has since unregistered between delegation start and an inner-loop tool call → the in-flight loop MUST get a clear in-band error. In practice this is the existing Tier-2 dispatch timeout ("no client executed the tool within Ns") rather than an instant "tool no longer available" detection — there is no separate live-unregistration-detection channel — but either way it MUST surface in-band, never hang or silently no-op.
- The delegating run is cancelled (user hits Stop) while an inner-loop delegation is in progress → the inner loop MUST settle the same way an in-flight top-level step does today (this falls out naturally from sharing the same run/signal, not a new mechanism).
- An agent's persisted `tools:` allowlist contains a mix of valid and invalid ids → the valid ones MUST still work; only the invalid ones are flagged. Partial failure is not total failure.
- Two windows of the same app are open simultaneously → surface-agent discovery MUST disambiguate by window (FR-022's auto-suffix), not just by app id; delegating to "the wrong one" must not be possible by construction.
- A surface agent's derived id collides with a **persisted** agent's id → registration MUST fail loudly (FR-022); it MUST NOT silently shadow the persisted agent.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: There MUST be exactly one tool-resolution registry (the v2 registry: `assistantTools()` / `gate.ts`) for every execution context — active/primary agent, and every kind of delegation (named, ephemeral, surface). The legacy sub-agent execution engine (`src/lib/agent/subagents/runner.ts`'s `runLocal`/`runToolLoop` path, `src/lib/agent/subagents/tools.ts`'s `ALL_TOOLS`/`toolsFor()`) MUST be retired, not merely deprecated.
- **FR-002**: An **ephemeral agent** MUST be definable inline at the point of delegation as `{name, systemPrompt, type}` (`type: "local"` or `"claude"`; `"claude"` additionally accepts today's `ephemeralSubagentType`), with no persisted file ever written for it and no `tools:` field accepted as part of its spec. FR-003/FR-004/FR-016/FR-017 (tool/skill/mcp inheritance, prompt handling) apply only when `type: "local"` — see FR-024 for `type: "claude"`.
- **FR-003**: A `type: "local"` ephemeral agent's resolved tool set MUST be computed as every tool in the delegating agent's `gate.allow` (its full configured allowlist — deferred status is irrelevant to eligibility, since FR-004 makes all inherited tools immediately visible regardless), filtered to `execution: "server"` only. This computation MUST have no code path that yields an unset/empty result by omission — the "no tools field means empty" ambiguity that produced the original zero-tools bug MUST NOT be reachable for ephemeral agents.
- **FR-004**: A `type: "local"` ephemeral agent's inherited tools MUST all be immediately visible (no deferred/discovery layer applied to inherited tools), regardless of whether the delegating agent itself had deferred or already revealed that tool.
- **FR-005**: A named (persisted `AGENT.md`) agent's `tools:`/`deferredTools:` configuration MUST continue to work exactly as today in shape and meaning; only its resolution path changes (unified registry instead of the legacy catalog).
- **FR-006**: A mounted app window MAY register a **surface agent** (see FR-009 for its full shape) via a mechanism analogous to `registerAppSurfaceTools`, scoped to that window's lifetime, and MUST unregister automatically on unmount.
- **FR-007**: A surface agent's `toolNames` MAY include Tier-2/frontend tools contributed by its own window; these MUST resolve correctly when the surface agent is delegated to, because delegation runs in the same run/browser-attachment as the delegating agent (see FR-011).
- **FR-008**: A surface agent's shape MUST be `{windowId, name, description, systemPrompt, toolNames}` — the app supplies its own `systemPrompt` directly as part of the registration payload (FR-006 covers the registration mechanism, not the shape). `description` MUST be present and SHOULD be a real, matchable one-line description — `find_agent` scoring MUST NOT rely solely on `name`.
- **FR-009**: A surface agent's id MUST be derived by slugifying its `name` (reusing the same slugification already used for ephemeral agent ids). See FR-022 for collision handling.
- **FR-010**: `agent_list` and `find_agent` MUST include currently-registered surface agents alongside the persisted agent roster. A surface agent MUST be discoverable by any run currently attached in the same browser client (tab) that has its window mounted — snapshotted at that run's start and live-pushed mid-run, exactly mirroring the existing surface-tools mechanism (not scoped to "the same BOS conversation" specifically, since surface tools already aren't either). Discovery results MUST flag which entries are window-scoped (vs. persisted), so a delegating agent understands a surface-agent target may disappear. A surface agent MUST stop being discoverable the instant its window unregisters **in any run started after that point**. An already-active run that snapshotted (or was mid-run-pushed) the surface agent before unregistration MAY still list it until that run ends — this mirrors today's surface-*tools* mechanism exactly (also additive-only within an active run: `run-manager.ts`'s `addSurfaceTools` never removes an entry mid-run either), so this spec is not lowering any existing bar. A stale in-flight reference resolves via the existing Tier-2 dispatch timeout (Edge Cases), not a live retraction.
- **FR-011**: Delegation (to a named, ephemeral, or surface agent) MUST execute as an **inner loop** that shares the delegating run's own dispatch mechanism (`emit`, `awaitFrontendResult`) — it MUST NOT create a new `Run`, a new conversation, or any new client-attachment/event-stream wiring.
- **FR-012**: The inner loop MUST start from a blank message history, seeded only with the `task` string supplied at the point of delegation. It MUST NOT receive the delegating conversation's transcript.
- **FR-013**: The inner loop's own steps (model turns, tool calls, tool results) MUST be reported as nested progress events (extending the existing `ctx.onEvent`/`encodeNested()` mechanism) and MUST NOT be persisted via `io.saveMessages()` as top-level conversation messages.
- **FR-014**: Regardless of how many steps the inner loop takes, the delegating run's persisted transcript MUST gain exactly one `tool_call` event and one `tool_result` event for the whole delegation.
- **FR-015**: If a named agent specifies a `model` override, its inner loop MUST use it. An ephemeral or surface agent (neither has a `model` field) MUST run on the delegating run's currently configured model.
- **FR-016**: A `type: "local"` ephemeral agent MUST additionally inherit the delegating agent's `skills` and `mcp` allowlists in full (the same rationale as FR-003's tool inheritance: a focused, temporary clone of what the parent itself can do, minus frontend tools). "In full" means inheriting the allowlist AS CONFIGURED, including unset/empty — which (per today's unchanged unset-means-all semantics) grants access to EVERY skill/MCP server, not zero; this MUST NOT be reinterpreted as "skip inheritance when unset," the same unset-vs-empty trap FR-003 was written to avoid for tools. Inheriting the allowlist only grants permission — using a skill in practice additionally requires `skill_list`/`skill_load` themselves to be present in the ephemeral agent's inherited *tool* set (FR-003), which they will be whenever the parent's own `gate.allow` includes them. A named agent's `skills`/`mcp` allowlists continue to gate skill-loading/MCP access exactly as today, now resolved through the unified registry.
- **FR-017**: `useDefaultPrompt` MUST NOT apply to ephemeral or surface agents — their system prompt (supplied explicitly at construction/registration time) is used verbatim, never prepended with the shared default-prompt template. A named agent's existing `useDefaultPrompt` behavior is unchanged.
- **FR-018**: The inner loop MUST have an explicit, finite step cap, independent of and never counted against the delegating run's own step budget (exact default(s) to be fixed in `plan.md`; MAY differ by delegation kind). On exhaustion, the inner loop MUST return a clear, in-band partial-completion summary (mirroring the existing step-limit message convention) as its result — never a silent truncation or a hang.
- **FR-019**: The **informational content** of the delegation's `tool_result` — the inner loop's own final natural-language answer — MUST be carried verbatim, with no truncation or re-summarization of that answer's own text. This does not preclude the existing structural envelope around it (today's `"[agent · type] N step(s)\n\n<output>"` header plus `encodeNested()`'s trailing marker payload, both there purely for UI/nested-card rendering, unchanged by this spec): "verbatim" governs the answer's own words, not the surrounding wrapper.
- **FR-020**: Any inner-loop failure — model/provider error, step-cap exhaustion (see FR-018), target agent not found, or depth-guard triggered (FR-024(b)) — MUST reach the delegating agent as an in-band error string (matching the existing in-band-error convention), and MUST NEVER surface as an unhandled exception or a silent no-op. The existing `contentOnly` input to `agent_delegate` is unaffected by this spec; it remains meaningful only for `type: "claude"` delegation (feature-branch gating), which this spec does not change.
- **FR-021**: Two delegations MUST NEVER be concurrently in-flight within the same run. This MUST hold by construction (the outer loop, and transitively every inner loop, executes its own tool calls strictly sequentially) — no additional serialization or event-correlation mechanism is to be added, since none is needed.
- **FR-022**: A surface agent's derived id (FR-009) that collides with an existing **persisted** agent's id MUST cause registration to fail, with the collision surfaced (per FR-023) — a surface agent MUST NEVER shadow a persisted agent. A derived id that collides with **another currently-registered surface agent's** id (e.g. two windows of the same app) MUST be automatically disambiguated (e.g. a short suffix derived from `windowId`). `agent_delegate`'s resolution order MUST check the persisted roster before the surface-agent registry.
- **FR-023**: Any tool id that fails to resolve — in a named agent's configured allowlist, or in a computed ephemeral/surface tool set, or a surface-agent id collision (FR-022) — MUST be surfaced, never silently dropped, per the standing no-silent-failures policy. For a named agent, this MUST be visible via an existing observability surface (e.g. Settings) in addition to a log line. For a surface agent (which has no Settings page), the primary surfacing venue MUST be a synchronous check against the persisted roster performed by the registering client **at registration time**, rejecting the registration in-band (visible to the registering app, not merely a server log) on collision. A second, server-side check at run-start is an accepted backstop ONLY for the race where a persisted agent is created *after* a surface agent already registered (a case the client-side check cannot see) — that backstop MAY be log-only, since there is no live channel back to the registering window at that point. Every "log line" in this FR is a real, central-logging-service record (see FR-027), not an ad-hoc `console.*` call that only the current process/browser tab can see.
- **FR-027**: Delegation's operational lifecycle MUST be recorded through BOS's existing central logging service (`017-central-logging`), not a new ad-hoc mechanism — this is a distinct concern from FR-023's user/developer-facing "surfacing" (Settings, registration-time rejection): FR-027 is about there being a durable, centrally-queryable trail at all. Specifically: (a) a delegation starting and finishing (tagged with its outcome — completed, step-cap exhausted, cancelled, or errored) MUST be logged; (b) a depth-guard rejection MUST be logged (today, hitting the legacy depth guard produces zero log trail — a silent-to-the-log failure this spec closes); (c) a named agent's unresolved tool/deferred-tool ids (FR-023) and a surface agent's registration-time or run-start collision (FR-022/FR-023) MUST be logged in addition to whatever in-band/UI surfacing they already get; (d) a server tool's own idle-timeout firing (ANY tool, not delegation-specific — a consequence of the linked-abort fix this spec's inner-loop model requires, see `plan.md`) MUST be logged, since today it is visible only as an in-band string to the model, with no operational trail for an unattended/headless caller. Log records MUST carry delegation metadata (agent id, delegation kind, depth, step count, outcome) but MUST NOT carry the full `task` string or tool arguments/results — mirroring the existing primary-run "run started"/"run finished" log shape, which already logs metadata only, never message content. Central logging remains fire-and-forget and non-fatal (per 017): a logging failure MUST NEVER affect delegation behavior or surface as an error to the user.
- **FR-024**: Two distinct rules, packed together because they share the same "ephemeral/claude" boundary condition: **(a)** FR-002–FR-004 and FR-016–FR-017 (ephemeral definition, tool/skill/mcp inheritance, prompt handling) apply to `type: "local"` delegation only — ephemeral `type: "claude"` delegation is unaffected by this spec and continues exactly as today via the Claude harness (`runClaudeAgent`), which provisions its own tools independently of the v2 registry. **(b)** Nested-delegation depth guarding (generalizing today's `MAX_DELEGATE_DEPTH`, currently enforced only inside the tool call site whose registered id is `dev_delegate` — see the Migration section's note on that name — top-level `agent_delegate` calls do not thread a depth value at all today) MUST be generalized to apply uniformly across named, ephemeral, and surface-agent delegation, regardless of `type`. (Rule (b) is NOT limited to `type: "local"`: a `type: "claude"` delegation still counts against the same depth budget, it simply never itself becomes an inner loop.)
- **FR-025**: Delegation intentionally MAY grant a target persona (named or surface) tools the delegating agent's own allowlist lacks — this is the deliberate capability-composition mechanism that fixes the motivating bug, not a bypass to be restricted. A built-in app's surface agent MAY name any registered server tool or its own Tier-2 tools in `toolNames`, with no additional restriction beyond what is already true of any BOS built-in code today. Extending this capability to sandboxed/installed (less-trusted) apps is explicitly out of scope and not assumed to be supported by this spec.
- **FR-026**: No delegation target (named, ephemeral, or surface) MAY retain state or message history across separate delegation calls. Each call MUST be independently blank-slate, with no exception carved out for "the same" surface agent being called twice.

### Key Entities

- **Agent** — id, name, description, `systemPrompt`, `type: local|claude`, `tools`/`skills`/`mcp`/`deferredTools`, `model?`, `useDefaultPrompt?`, and (for `type: "claude"`) `subagentType`. Stored under `data/agents/<id>/AGENT.md`. Now resolved through the unified registry in every context (FR-001, FR-005, FR-015–FR-017).
- **Ephemeral Agent Spec** — `{name, systemPrompt, type}` plus, for `type: "claude"` only, `ephemeralSubagentType`. Constructed inline at delegation time, never written to disk. For `type: "local"`: tool/skill/mcp set always derived from the delegating agent (FR-003, FR-016), never separately configured; no `model` field (FR-015); no `useDefaultPrompt` behavior (FR-017). For `type: "claude"`: unaffected by this spec (FR-024(a)).
- **Surface Agent** — `{windowId, name, description, systemPrompt, toolNames}`, registered by a mounted app window, live only for that window's lifetime, discoverable via `agent_list`/`find_agent` while registered (FR-009, FR-010), with automatic id-collision handling (FR-022).
- **Delegation Inner Loop** — the execution unit for any delegation target. Not a `Run`, not a persisted `Conversation` — a nested tool-loop instance that reuses the parent run's `emit`/`awaitFrontendResult`, reports its own steps as nested progress, has its own step cap (FR-018), and returns its final answer verbatim as the delegation's `tool_result` (FR-019).
- **Nested Progress Event** — existing concept (`ctx.onEvent`, `encodeNested()`), generalized as the sole channel for an inner loop's intermediate activity; never conflated with top-level persisted messages.
- **Discovery Result** (`agent_list`/`find_agent` entry) — `{id, name, type, description, score, scope: "persisted" | "surface"}` — extends today's shape with `scope` (FR-010) so a delegating agent knows a given target may be transient.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Delegating to any agent (named, ephemeral, or surface) resolves tools from the single unified registry; zero tool ids silently vanish between "configured" and "actually callable."
- **SC-002**: A `type: "local"` ephemeral agent has access to exactly the delegating agent's current allowed server-executable tools (plus its skills/mcp allowlists), verified by successfully invoking at least one such tool, and runs on the delegating run's model.
- **SC-003**: A surface agent's Tier-2 tool (e.g. `ui_preview_render`) is successfully callable when delegated to, verified end-to-end (the mockup actually renders in the live window).
- **SC-004**: A surface agent is not discoverable via `find_agent`/`agent_list` once its window has closed.
- **SC-005**: A multi-step (5+) delegation results in exactly one persisted `tool_call`/`tool_result` pair in the delegating conversation's saved transcript; the inner steps are visible only as nested/expandable progress.
- **SC-006**: Two sequential delegations to the same surface agent, with no shared state between them, still produce correct behavior, because the delegating agent supplied adequate context in each task string.
- **SC-007**: No agent-configuration tool id, and no surface-agent id collision, ever fails silently — every such case is surfaced somewhere a developer or user can see it.
- **SC-008**: An inner loop that exhausts its step cap returns a clear partial-completion summary as its `tool_result`, never a hang or an unhandled error.
- **SC-009**: Two windows of the same app registering surface agents with the same configured `name` both remain independently, correctly delegatable (no id collision, no ambiguity).
- **SC-010**: Every delegation's start/finish (with outcome), every depth-guard rejection, every unresolved-tool-id/collision case, and every server-tool idle-timeout is queryable afterward in Settings → Logs (component `assistant.delegate`/`assistant.surface-agents`/`assistant.tools`, filterable by conversation) — not just visible in the moment (in-band error, console) and then gone.

## Examples

*(Non-standard section, included because these nuances are easy to get subtly wrong.)*

### Example 1 — Ephemeral agent tool/skill/mcp inheritance, including the deferred case

The "assistant" agent (primary, currently active) has, among its configured allowlist (`gate.allow`): `memory_search` (server), `gmail_messages_list` (server, currently **deferred and not yet revealed** — the assistant hasn't called `find_tools` for it this conversation), `bos_app_launch` (frontend-only), `ui_preview_open` (frontend-only, deferred). Its `skills` allowlist includes `recall-long-term-memory`; its `mcp` allowlist includes `playwright`.

It delegates: `agent_delegate({ ephemeralName: "Quick Researcher", ephemeralSystemPrompt: "You research a topic and summarize findings.", ephemeralType: "local", task: "Summarize recent news about X" })`.

Per FR-003, the ephemeral agent's resolved tool set is every tool in the assistant's `gate.allow` filtered to `execution: "server"` — **including `gmail_messages_list`**, even though the assistant itself hadn't yet revealed it via `find_tools`. Deferred status governs only what the *assistant* sees; it has no bearing on ephemeral-inheritance eligibility (FR-003), and per FR-004 the ephemeral agent sees it immediately, with no deferred layer of its own. **Not** included: `bos_app_launch`, `ui_preview_open` — both frontend-only, excluded per the deliberate ephemeral-agent scope line (FR-003), not because the inner loop lacks the attachment to run them: Tier-2 tools ARE reachable through delegation, but only via a **surface agent**'s bounded, app-declared `toolNames` — never via an ephemeral agent's open-ended "inherit everything the parent can do." Per FR-016, the ephemeral agent also inherits the `recall-long-term-memory` skill and `playwright` MCP access (and, per FR-016's clarified wording, would inherit ALL skills/MCP servers if the assistant's own allowlists were unset rather than these specific entries). Per FR-015, it runs on the assistant's currently configured model (it has none of its own).

### Example 2 — Surface agent lifecycle

1. User opens the UI Preview app. Its mounted component registers a surface agent: `{name: "Generative UI Agent", description: "Specialist in rendering and iterating on live A2UI mockups in this UI Preview window.", toolNames: ["a2ui_render", "ui_preview_render"]}` (system prompt supplied separately, per FR-006/FR-009).
2. The primary agent calls `find_agent({query: "generate a UI mockup"})` → now returns `{id: "generative-ui-agent", name: "Generative UI Agent", type: "local", description: "...", score: ..., scope: "surface"}` (previously: nothing, since UI Preview tools weren't even in the primary's own allowlist, let alone exposed as a delegate target). The `scope: "surface"` flag tells the primary this target is transient.
3. The primary agent delegates: `agent_delegate({ agent: "generative-ui-agent", task: "Build a two-tab UI: Register (form) and Subscription (3 pricing panels)..." })`.
4. The inner loop runs with the app-supplied system prompt and exactly `{a2ui_render, ui_preview_render}` as its tools — both resolve and dispatch correctly, because the inner loop shares the SAME run's `awaitFrontendResult`, and the SAME browser tab (already attached to this run, already has the UI Preview window mounted) answers them exactly as if the primary agent had called them directly.
5. The delegating conversation's transcript gains one `tool_call` (`agent_delegate`, the task) + one `tool_result` (the inner loop's own final answer, verbatim per FR-019, e.g. "Built the two-tab mockup with Register and Subscription tabs.") — not the 6+ raw turns the inner loop actually took.
6. User closes the UI Preview window. `find_agent`/`agent_list` no longer return "Generative UI Agent." A subsequent `agent_delegate({agent: "generative-ui-agent", ...})` fails with a clear "no such agent" error (FR-020).

### Example 3 — Transcript shape across repeated delegation (no cross-call memory)

Conversation shape: `U→A→U→A→D₁(UIPreview)→A→U→A→D₂(UIPreview)→A→U`.

- After `D₁` completes, `A`'s persisted transcript contains (among the earlier `U`/`A` turns): one `tool_call` ("delegate to Generative UI Agent: build a two-tab mockup...") + one `tool_result` ("Built the two-tab mockup...").
- `D₂`'s inner loop starts **completely blank** — it has never heard of `D₁`, does not know a mockup already exists, and receives only whatever `task` string `A` gives it this time.
- But `A` itself, composing that second task, has full access to its own transcript — including `D₁`'s tool-call+result — so it correctly writes something like: `task: "The mockup currently has a Register tab (Name/Email/Password/Next) and a Subscription tab (3 pricing panels: Free/Pro/Ultimate). Make the pricing panel buttons larger and add a 'Most Popular' badge to Pro."` `D₂` succeeds despite having no memory, because `A` — not `D` — is the one who supplies continuity.

### Example 4 — Silently-failing tool id, now surfaced

An agent's `AGENT.md` lists `tools: [..., gmial_messages_list, ...]` (typo). Today: `toolsFor()`'s `allowed.filter((id) => ALL_TOOLS[id])` drops it with zero signal — the agent simply never has Gmail access and nothing ever says why. Under this spec (FR-023): agent-load/gate-construction validates every configured id against the unified registry; `gmial_messages_list` doesn't resolve, and this is surfaced (logged, and visible in Settings against that agent) — while every *other*, correctly-spelled tool in the same list still works normally (a partial, not total, failure).

### Example 5 — Depth guard, generalized, applying to ephemeral agents

The "assistant" agent has `agent_delegate` itself in its server-tool set. It spins up an ephemeral agent, which — per FR-003 — inherits `agent_delegate` too. If that ephemeral agent then calls `agent_delegate` to spin up *another* ephemeral agent, the generalized depth guard (FR-024(b)) must cap this — unlike today, where the depth parameter is only threaded and checked inside the tool call site registered as `dev_delegate` specifically, and a top-level `agent_delegate` call doesn't participate in depth-tracking at all. This spec requires the guard to apply uniformly, regardless of whether the chain is named agents, ephemeral agents, surface agents, or a mix.

### Example 6 — Inner loop exhausts its step cap

A delegation's inner loop hits its step cap before producing a final answer. Per FR-018, it does not hang, error opaquely, or silently return whatever partial text it has — it returns an explicit summary in the same spirit as the existing v2 loop's own step-limit message (e.g. "Reached the step limit before finishing this sub-task. Partial work may already be applied — review what was done."), and this becomes the delegation's `tool_result` (FR-019, FR-020), so the delegating agent can see exactly what happened and decide whether to retry, adjust the task, or continue manually.

### Example 7 — Surface-agent id collision

Two UI Preview windows are somehow open at once, both registering a surface agent named "Generative UI Agent." The second registration's naive slugified id (`generative-ui-agent`) collides with the first's. Per FR-022, this is auto-disambiguated (e.g. the second becomes `generative-ui-agent-2`, using a short suffix derived from its `windowId`) — both remain independently discoverable and delegatable, and the collision is NOT surfaced as an error (unlike a collision against a persisted agent, which is always an error). Separately: if a third-party app tried to name its surface agent `"assistant"` (colliding with the persisted default Assistant), registration is rejected outright and surfaced (FR-022, FR-023) — a surface agent may never shadow a persisted one.

## Assumptions & Dependencies

- Builds on `016-unified-agents`'s framing ("one Agent concept") and `013-build-studio-agentic`'s surface-tools mechanism (`registerAppSurfaceTools`, the live-push-into-run pattern), extending the latter with an agent-shaped sibling.
- Assumes the v2 server-owned-run architecture (`src/lib/assistant/agent-loop.ts`, `run-manager.ts`, `registry.ts`) as the sole execution substrate. `016` predates this rewrite and still describes a CopilotKit-era dual-engine world; this spec is what finishes `016`'s original intent under the architecture that actually shipped.
- Out of scope for this spec: true cross-run context isolation for delegation (a separate run whose events a browser tab additionally attaches to). Rejected in favor of the in-run inner-loop model: it avoids new cross-run event-relay engineering, and full history isolation was judged not to matter for a same-user, same-window, tightly-coupled workflow like iterative UI design — may be revisited separately if isolation ever proves necessary in practice.
- Out of scope: giving a delegation target its own persisted, cross-call memory/scratch-state. Explicitly rejected (FR-026) — if ground-truth drift ever becomes a real problem, the fix is a "read current live state" query tool, not delegate-side memory; not designed here.
- `type: "claude"` agents (routed to the Claude Code harness, `runClaudeAgent`) are unaffected by this spec's tool-registry unification for either named or ephemeral use (FR-024(a)) — that harness has always had its own, separate tool-provisioning model (the harness's own CLI/MCP tool access), which is out of scope here.
- Extending surface-agent registration to sandboxed/installed (less-trusted) apps is explicitly not assumed or designed for here (FR-025); if that capability is ever wanted, it requires its own trust-boundary review first.
- The exact inner-loop step-cap default(s), and the exact mechanism for surfacing a **named** agent's FR-023 violations (log line, Settings badge, or both), are left to `plan.md` — this spec fixes the *properties* those choices must satisfy, not the literal numbers/UI. (A **surface** agent's FR-023 surfacing mechanism is now fixed by the FR itself: a registration-time check against the persisted roster, not left open.)
- FR-027's "log line" is not a new logging system — it assumes and reuses BOS's existing central logging service (`017-central-logging`: `logger()`/`clog()`, component-tagged, conversation-scoped, fire-and-forget, viewable in Settings → Logs). No new logging infrastructure is in scope here; only which events this feature emits through it. Today's legacy delegation engine (`subagents/runner.ts`) logs NOTHING at all — not even its own silently-swallowed errors (e.g. `runner.ts:147-148`'s bare `catch { /* ignore staging errors */ }`) — so this is a real, not cosmetic, gap FR-027 closes.

## Migration / touchpoints (for sizing)

- **`src/lib/agent/subagents/tools.ts`** — `ALL_TOOLS`, `toolsFor()`, `STATIC_SCHEMAS`: retired. Audit needed for FR-001: known legacy-only candidates to check against the v2 registry before retiring include the **scheduler tools** (`SCHEDULER_TOOLS`) and the **spec-tools availability rule** (the legacy path hands spec tools to any agent via `makeSpecTools`; the v2 registry currently gates spec tools to Build Studio-style agents — this behavioral difference must be reconciled explicitly, not just carried over silently). The audit also needs to walk existing `data/agents/*/AGENT.md` and `seed/agents/*/AGENT.md` for ids whose resolution changes or has already silently broken — e.g. `data/agents/build-studio/AGENT.md`'s `tools:` list today mixes current ids with several pre-016 CopilotKit-era camelCase action names (`listSpecs`, `openSpecArtifact`, `delegateToSubAgent`, …) that already silently fail to resolve under `toolsFor()`'s filter — this spec's validation (FR-023) will surface these for the first time, so the audit should clean them up rather than merely let them get flagged.
- **`src/lib/agent/subagents/runner.ts`** — `runLocal`/`runToolLoop` path retired; `runSubAgent` for `type: "local"` re-routes through the new inner-loop mechanism instead. `type: "claude"` path (`runClaudeAgent`) is untouched (FR-024(a)).
- **`src/lib/assistant/tools/server/subagents.ts`** — `agent_delegate`'s `execute()` rewritten to build and run an inner loop (FR-012–FR-014, FR-018–FR-020) instead of calling the legacy `runSubAgent`; ephemeral-spec construction updated per FR-002–FR-004, FR-015–FR-017 (no `tools`/`model` field, always-derived tool/skill/mcp set, no default-prompt prepending); `contentOnly` and `ephemeralSubagentType` handling preserved unchanged for the `type: "claude"` path. The tool call site today referred to in comments/docstrings as "the `delegate_to_developer` tool" is registered under the id **`dev_delegate`** (`subagents/tools.ts`'s `DELEGATE_TO_DEVELOPER = "dev_delegate"` constant) — every seeded skill and `AGENT.md` that references it already uses `dev_delegate` (e.g. `seed/agents/build-studio/AGENT.md`, `data/skills/bos-app/SKILL.md`). A v2-native replacement for this tool MUST keep the id `dev_delegate` — it is not a rename, just completing an already-established, already-referenced-everywhere id that has no v2 implementation yet.
- **`src/lib/assistant/agent-loop.ts`** — needs the actual inner-loop execution primitive (reusable by `agent_delegate`): run N steps (own cap, FR-018) against a given system prompt + tool set + blank history, reporting via `onEvent` instead of `emit({type:"message"})`/`io.saveMessages()`, returning its final answer verbatim (FR-019) or an in-band failure/exhaustion summary (FR-018, FR-020).
- **Depth guard** — currently `MAX_DELEGATE_DEPTH` is checked only inside the `dev_delegate` tool implementation (`runner.ts`); top-level `agent_delegate` calls do not thread a depth value today. FR-024(b) requires this to be generalized so depth is tracked and enforced uniformly for every delegation path (named/ephemeral/surface), including the new inner-loop mechanism.
- **`src/lib/assistant/client/surface-tools.ts`** — needs an agent-registration sibling to `registerAppSurfaceTools`/`getActiveSurfaceToolDeclarations` for surface agents (FR-006, FR-009), including the same live-push-into-run mechanism `run-client.ts` already uses for surface tools, plus id-collision handling (FR-022).
- **`src/lib/assistant/run-manager.ts`** — needs a `Run`-scoped registry for currently-known surface agents (paralleling `addSurfaceTools`), so `agent_list`/`find_agent` (server-side) can see them, with the `scope` flag (FR-010).
- **`src/lib/assistant/tools/server/discovery.ts`** (`find_agent`) and the `agent_list` tool — must merge the persisted roster with currently-registered surface agents, extending the result shape with `scope` (FR-010) and matching against the new `description` field (FR-008).
- **`src/apps/ui-preview/`** — first real surface-agent registration, as the motivating example (Example 2).
- **Validation/surfacing (FR-023)** — for named agents, lands in `src/lib/agent/subagents/store.ts` (agent load/save path) and/or `src/lib/assistant/gate.ts` (gate construction). For surface agents, the primary check is client-side at registration time (a synchronous lookup against the persisted roster, e.g. via the existing `GET /api/subagents`), with a server-side run-start recheck as the backstop for the create-after-register race (see `plan.md`'s Phase 4 for the concrete mechanism).
- **Logging (FR-027)** — server-side events (delegation start/finish, depth-guard rejection, named-agent tool-id validation, surface-agent run-start collision backstop, server-tool idle-timeout) go through `logger()` (`@/lib/logging`), reusing the existing `assistant.run`/`assistant.hooks`-style component-naming convention (new components: `assistant.delegate`, `assistant.surface-agents`, `assistant.tools`). The one client-side event (a surface agent's registration-time collision rejection) goes through the existing browser logger (`clog()`, `src/lib/logging/client/browser-logger.ts`) — it already ships to the SAME central timeline (`/api/logs` → Supervisor or local file sink), so this needs no new ingestion endpoint. See `plan.md` for exact call sites.
