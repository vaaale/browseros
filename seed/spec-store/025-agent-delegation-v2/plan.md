# Implementation Plan: Agent Delegation v2 (025)

**Branch**: `025-agent-delegation-v2` | **Date**: 2026-07-13 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification `specs/bos-system-specs/025-agent-delegation-v2/spec.md` (delta against `016-unified-agents`, all independent-review findings resolved ‚Äî see `spec-review.md`).

## Summary

Retire the legacy sub-agent tool-resolution engine (`src/lib/agent/subagents/tools.ts` + `runner.ts`'s `runLocal`) and make every delegation ‚Äî named, ephemeral, or window-scoped surface agent ‚Äî resolve tools from the same v2 registry (`assistantTools()`/`gate.ts`) the primary personality uses, execute as an **inner loop**, and collapse to one `tool_call`+`tool_result` pair in the parent transcript.

The central implementation finding from grounding this plan in the current code: **the inner loop does not need a new execution primitive.** `runAgentLoop` (`src/lib/assistant/agent-loop.ts`) is already a pure, fully-injected function ‚Äî `tools`, `gate`, `maxSteps`, `emit`, `awaitFrontendResult`, `io` (load/save messages), and `signal` are all constructor-injected, not baked in. A delegation can call `runAgentLoop` a second time, in-process, with: an in-memory `io` (blank history, FR-012), an `emit` that forwards to `ctx.onEvent` instead of the parent's SSE stream (nested progress, FR-013), the delegate's own `tools`/`gate`/`maxSteps`, and the **same** `awaitFrontendResult`/`signal` the parent run already has ‚Äî which is exactly why a surface agent's Tier-2 tools (FR-007) and mid-delegation Stop (Edge Cases) work for free, with no new wiring. `runAgentLoop`'s existing step-limit message (`STEP_LIMIT_TEXT`) also already satisfies FR-018's exhaustion-summary requirement verbatim ‚Äî no new "partial completion" text needs to be authored.

This shrinks the work to four real phases: (1) a small `inner-loop.ts` wrapper + gate/prompt builders per delegation kind, (2) rewiring `agent_delegate` (and a new `dev_delegate`) onto it with a generalized depth guard, (3) retiring the legacy engine and porting the handful of tools that exist ONLY there today, (4) surface agents (client registry + discovery merge + id collisions).

E2e: Playwright specs under `e2e/`, request-fixture style for API/transcript-shape assertions (mirrors `e2e/run-events-replay.spec.ts`) or browser-fixture style for UI assertions (mirrors `e2e/013-ui-preview.spec.ts`), driven by the existing scripted provider (`BOS_E2E_SCRIPTED=1` + `@@e2e {"turns":[...]}`, `e2e-provider.ts`) for determinism ‚Äî reused for the inner loop's own model calls too (see Current-state audit). ## Current-state audit (grounds the phasing below)

Verified directly against the code on this branch before writing this plan (not assumed from the spec's own Migration section, which was written earlier and turned out to be more pessimistic than reality in a few places):

- **`runAgentLoop`** (`agent-loop.ts:148`) is already the reusable primitive ‚Äî see Summary. `AgentLoopDeps` (`agent-loop.ts:54-74`) takes `tools`, `gate`, `maxSteps`, `emit`, `awaitFrontendResult`, `io`, `signal`, `composeSystem`, `streamTurn` as plain fields; nothing here is `start-run.ts`-specific.
- **`agent_delegate`** (`tools/server/subagents.ts:56-117`) already calls `runSubAgent(def, task, {...})` with **no `depth` passed** (confirms A4/FR-024(b)'s finding precisely) and already wraps its result in `encodeNested()` (`nested-events.ts`) ‚Äî this wrapping stays unchanged; only what runs *inside* changes.
- **Live nested progress already has a conduit**: `ToolContext.onEvent` (`tools.ts:19-26`) is wired by `runServerTool` (`agent-loop.ts:96-141`) to `emit({ type: "tool_progress", callId, event })` ‚Äî i.e. every `ctx.onEvent(x)` call a server tool makes already becomes a live SSE event scoped under that tool call's `callId`. FR-013's "nested progress" requirement needs **no new transport** ‚Äî the inner loop's `emit` just needs to call `ctx.onEvent(event)` instead of writing to the parent run's top-level stream.
- **The legacy engine has partially converged already**: `subagents/tools.ts`/`runner.ts` picked up their own deferred/reveal mechanism (`pickDeferredIds`, `makeDiscoveryTools`, `effectiveDeferred`) and a `find_agent`/`find_tools` pair, independently of the v2 registry's `gate.ts`/`discovery.ts`. These are two working, near-identical implementations of the same idea (visible in code comments referencing "(025)" ‚Äî that was a *different*, already-shipped feature, "deferred-tool-discovery" on branch `bos/deferred-tools-discovery`, which happened to reuse the number "025" informally in commit messages before this spec claimed it in the spec-store; no actual spec-store collision, since no `025-deferred-tool-discovery/spec.md` was ever committed ‚Äî but worth knowing so the two aren't confused during implementation). **This means FR-001's retirement is not "add discovery to the legacy path", it already has it ‚Äî it's a straight deletion once delegated execution stops depending on it.**
- **`devSourceTools()` (v2) already imports from the legacy file**: `tools/server/dev-source.ts:4` does `import { DEV_TOOLS } from "@/lib/agent/subagents/tools"` and adapts it via `adaptLlmTools()` (`tools/server/adapt.ts`) ‚Äî so `subagents/tools.ts` cannot simply be deleted; `DEV_TOOLS`'s three tiny wrappers (`bos_source_list/read/search`, over `repo-fs.ts`) must move to a v2-native module first.
- **`run_command` already has full parity**: v2's `runCommandTools()` (`tools/server/run-command.ts:41`) uses `sessionKey: ${ctx.conversationId}:${ctx.agentId}`, equivalent to legacy's per-(session,agent) key (`runner.ts:55-56`). No gap ‚Äî legacy's `makeRunCommandTool` is dead code once `runLocal` is deleted.
- **Scheduler tools are legacy-only today** (`SCHEDULER_TOOLS`, `subagents/tools.ts:9,112,245`) ‚Äî genuinely absent from `assistantTools()`. Confirmed by grep: no scheduler tool appears anywhere under `src/lib/assistant/`. This is the one real, unaddressed capability gap FR-001's audit needs to close, not just a resolution-path change.
- **`dev_delegate` (`DELEGATE_TO_DEVELOPER`) is legacy-only and per-run-factory-built** (`runner.ts:27-50`, `makeDelegateTool`) ‚Äî needs a genuine v2-native replacement, not a port, since its old shape (closure over `onEvent`/`depth`/`featureBranch`) is superseded by the same inner-loop + depth-guard mechanism `agent_delegate` gets.
- **VFS tools (`file_list/read/write/mkdir`) are `execution: "frontend"` in v2** (`tools/frontend-declarations.ts:48-64`) but **server-side** in the legacy engine (`subagents/tools.ts:20-49`, direct `@/os/vfs` calls). Because ephemeral agents inherit only `execution: "server"` tools (FR-003, deliberately ‚Äî see the corrected Clarifications rationale), an ephemeral agent gets no VFS access under this design. This is NOT a regression that needs a server-side VFS port: a **named** agent's `tools:` allowlist can include `file_read`/`file_write` today, and it dispatches correctly through the shared `awaitFrontendResult`, exactly like a surface agent's own tools. Decision: **do not port VFS tools to a server-side form** ‚Äî an ephemeral agent needing file access is the wrong tool for the job; delegate to a named agent instead. Flagged explicitly here so it isn't silently discovered as a gap during implementation.
- **No per-call model override exists anywhere today**: `streamModelTurn` (`model-turn.ts:22`) always calls the parameterless `getProviderConfig()` (`provider.ts:46`), which returns the single Settings-configured provider/model ‚Äî the primary run, and every existing delegated run, always use the same model. FR-015 ("if a named agent specifies `model`, its inner loop MUST use it") is therefore a **new capability**, not a resolution-path change ‚Äî scoped in Phase 1 below.
- **Agent roster is already unified**: both the legacy `find_agent` (`subagents/tools.ts:391-423`) and v2's `find_agent` (`discovery.ts:43-59`) read the same `listSubAgents()` (`subagents/store.ts:147`). Only the *tool* catalog forks today, not the *agent* catalog ‚Äî good news for Phase 4 (surface agents only need to be merged into the v2 side).
- **The e2e scripted-provider hook is a pure, reusable function, not `start-run.ts`-specific**: `e2eScriptedTurn(message: string): StreamTurn | null` (`src/lib/assistant/e2e-provider.ts:52`) takes an arbitrary string (gated on `BOS_E2E_SCRIPTED=1` + an `@@e2e {"turns":[...]}` prefix) and returns a scripted `StreamTurn` closure, or `null` to fall through to the real provider ‚Äî `start-run.ts:117` uses it as `e2eScriptedTurn(opts.message) ?? streamModelTurn`. **This is directly reusable for `runInnerLoop`**: calling `e2eScriptedTurn(task) ?? streamModelTurn` with the delegation's own `task` string gives deterministic, multi-step-scriptable e2e testing of delegation with zero new test infrastructure ‚Äî this is what makes Phase 3/5's e2e tasks (`tasks.md`) actually writable at all, since this repo's own e2e tests explicitly never assert on live-LLM output (`e2e/013-ui-preview.spec.ts`'s comment).
- **UI Preview is a `singleton: true` app** (`src/apps/ui-preview/manifest.ts:9`) ‚Äî two real windows of it can never coexist. Example 7 / US-4 acceptance scenario 4 (two windows registering same-named surface agents) is therefore **not reproducible as a true browser e2e test** for this app; it must be a unit test that calls `registerSurfaceAgent` directly with two different `windowId`s, bypassing the OS-level singleton constraint entirely (which is fine ‚Äî the behavior under test is the client registry's id-disambiguation logic, not the app's window management).
- **This repo has no wired unit-test runner** (no `vitest`/`jest` in `package.json`, confirmed by grep) ‚Äî existing "unit tests" (e.g. `src/lib/agent/scratchpad/__tests__/handlers.test.ts`) are hand-run, `assert()`-based files with a `runAll()` entry point, executed ad-hoc (e.g. via `tsx`), not through a test framework. New unit tests in this plan (`inner-loop.ts`, the linked-abort fix, surface-agent id-collision) follow this SAME convention, not a new one.
- **The legacy delegation engine logs NOTHING today** (FR-027): grepped `subagents/{runner,tools,store}.ts` for `logger()`/`console.*` ‚Äî zero hits in any of the three files. Confirmed concrete silent-failure sites: `runner.ts:147-148`'s bare `catch { /* ignore staging errors */ }`, and `runner.ts:176-177`'s outer catch, which returns `{...error: e.message}` in the result object but never logs ‚Äî a failed delegation leaves no trace in `data/logs` at all. Meanwhile v2's own `assistant.run`/`assistant.hooks` components (`start-run.ts:98-104,140`) already follow the central-logging conventions (`017-central-logging`) properly. This asymmetry is exactly why FR-027 exists ‚Äî it's not a nice-to-have bolted onto this migration, it's closing a real observability hole the legacy engine has always had.
- **The logging infrastructure this plan needs already exists in full, both sides**: server-side `logger().log/.info/.warn/.error(component, msg, data?, err?)` (`@/lib/logging`, dotted `<area>.<subarea>` component convention, e.g. `assistant.run`); client-side `clog(level, component, msg, data?, err?, conversation?)` (`src/lib/logging/client/browser-logger.ts:52`), which already batches and ships to the SAME central timeline via the existing `POST /api/logs` endpoint (`src/app/api/logs/route.ts`) ‚Äî no new ingestion endpoint is needed for the one client-side log call this plan adds (Phase 4's registration-time collision).

## Design

### Phase 1 ‚Äî Shared primitives

**`src/lib/assistant/inner-loop.ts` (new)**

```ts
interface InnerLoopSpec {
  systemPrompt: () => Promise<string>;   // composeSystem, per delegation kind (below)
  gate: ToolGateConfig;
  model?: string;                        // named-agent override only
}
async function runInnerLoop(
  parentRun: Run, ctx: ToolContext, spec: InnerLoopSpec, task: string, maxSteps: number,
): Promise<{ output: string; error?: string; steps: number; toolCalls: { tool: string; input: unknown }[] }>
```

- **`io`**: in-memory. `loadMessages()` ‚Üí `[]` (FR-012: blank slate). The `task` string enters exactly like a top-level user message: `runAgentLoop`'s own `AgentLoopInput.userMessage` field appends it (`agent-loop.ts:157-170`) ‚Äî `runInnerLoop` does not need to seed `messages` itself, just call `runAgentLoop({...}, { userMessage: { content: task } })`. `saveMessages(messages)` captures the latest array into a closure variable ‚Äî never touches `conversationIO`/disk. After `runAgentLoop` resolves, read the captured array back to extract:
  - `reason: "completed"` ‚Üí last `role: "assistant"` message's `content`, verbatim (FR-019's informational-content requirement ‚Äî no new truncation step; the surrounding `"[agent ¬∑ type] N step(s)\n\n<output>"` + `encodeNested()` wrapping is applied by `agent_delegate`/`dev_delegate` in Phase 2, not here).
  - `reason: "max_steps"` ‚Üí the loop already appended `STEP_LIMIT_TEXT(maxSteps)` as that same last assistant message ‚Äî **use it as-is** (FR-018's "mirrors the existing step-limit message convention" is met by literally reusing the function, not writing a new one).
  - `reason: "cancelled"` ‚Üí return whatever partial output exists (there won't be a final assistant message; return a fixed "Delegation cancelled." in-band string) ‚Äî note `runServerTool`'s OWN abort listener for the outer `agent_delegate`/`dev_delegate` call usually settles first with "Cancelled by user." on a real Stop, so this string is a defensive fallback that may rarely surface in practice, not the primary path (harmless either way ‚Äî both are in-band).
  - `reason: "error"` ‚Üí `result.error`, surfaced as `Error: agent_delegate: <error>` (FR-020).
- **`emit`**: **not** a raw forward of every `RunEventInput`. Shape it down to the legacy `ToolEvent = {tool, input}` pair the existing UI already renders live: `(e) => { if (e.type === "tool_call") ctx.onEvent({ tool: e.name, input: safeParseJson(e.args) }); }` ‚Äî ignore `step_started`/`text_delta`/`message`/`tool_result` noise (a raw per-character `text_delta` forward would spam the card). **Verified against the frontend, not assumed**: `ToolCallCard.tsx`'s live-progress path (`call.progress` ‚Üí `toNestedEvents` ‚Üí `NestedEventList` with `running` spinner, `ToolCallCard.tsx:35-40,82,118-122`) already exists and already expects exactly this `{tool, input}` shape ‚Äî it's the SAME rendering path `parseNested()`'s post-hoc payload uses. **This closes what would otherwise be an unscoped UI requirement (US-3.2's live nested tree): no new frontend code, just correct event shaping on the way in.**
- **`tools`**: `parentRun.tools` ‚Äî the SAME object reference the parent run mutates via `addSurfaceTools` (`run-manager.ts:180`), so a surface agent's Tier-2 tools (registered on the SAME window during THIS run) are visible immediately (FR-007).
- **`awaitFrontendResult`**: `(callId, ms) => runManager().awaitFrontendResult(parentRun, callId, ms)` ‚Äî literally the parent run, so Tier-2 dispatch lands on the same attached browser tab (FR-007, FR-011).
- **`signal`**: `ctx.signal` ‚Äî **not** `parentRun.abort.signal` directly (see "Linked abort" below for why this distinction matters). Because `ctx.signal` is fixed (below) to be a per-call signal that aborts on EITHER a real run Stop OR the outer `agent_delegate`/`dev_delegate` call's own idle-timeout firing, using it here means the inner loop stops promptly in both cases ‚Äî a genuine Stop still cascades down exactly like today (Edge Case), with no new mechanism needed for that part.
- **`streamTurn`**: `e2eScriptedTurn(task) ?? streamModelTurn` (`e2e-provider.ts:52`, reused verbatim ‚Äî same one-liner pattern `start-run.ts:117` already uses with `opts.message`). This is what makes deterministic e2e testing of delegation possible at all: an `@@e2e {"turns":[...]}` string passed as the delegation's `task` scripts the INNER loop's model turns, independently of whatever scripted (or real) provider the OUTER run is using. No new test infrastructure ‚Äî purely reusing an already-exported, already-pure function with a different input string.
- **`maxSteps`**: `spec`'s own cap, independent of the parent's `DEFAULT_MAX_STEPS` (FR-018). Default: reuse today's `DEV_MAX_STEPS = 40` for any delegation whose tool set includes dev/spec-style tools (mirroring `runner.ts:94-95`'s `isExtended` heuristic, ported as-is), else a smaller default (12, informed by but not bound to today's `MAX_STEPS = 8`/`DEFAULT_MAX_STEPS = 24` ‚Äî final number is a plan-level judgment call, not user-facing enough to need its own FR debate).
- **`toolTimeoutMs`**: the inner loop's OWN nested server-tool calls (e.g. the delegate itself calling `a2ui_render`) need a value too ‚Äî add a `toolTimeoutMs: number` field to `Run` (`run-manager.ts`), set once in `start-run.ts` alongside `run.tools`, and have `runInnerLoop` read `parentRun.toolTimeoutMs` and pass it straight through to its own nested `runAgentLoop` call. Reusing the same value the parent resolved is simplest and avoids inventing a second Settings knob.
- **`hooks`**: `undefined` ‚Äî no interception inside a delegation.

**Linked abort ‚Äî the outer idle-timeout must not orphan a running inner loop**: today, `runServerTool` (`agent-loop.ts:96-141`) hands every server tool's `execute()` the RAW run signal (`ctx: { signal, conversationId, agentId }`, where `signal` is `deps.signal` = `run.abort.signal`) ‚Äî its own idle-timeout firing only resolves `runServerTool`'s promise, it does NOT touch that shared signal. For an ordinary (fast, synchronous) server tool this is harmless; for `agent_delegate`/`dev_delegate`, whose `execute()` runs a WHOLE NESTED `runAgentLoop` that can legitimately run long, it means a single silent stretch (no `ctx.onEvent` call for the full configured timeout ‚Äî e.g. a local model server buffering with zero streamed deltas) makes the OUTER call settle with an in-band timeout error while the inner loop **keeps running detached**, possibly still driving `ui_preview_render` after the parent already reported failure. Fix, scoped to `runServerTool` itself (a small, general improvement, not delegation-specific): construct a per-call `AbortController` there, linked to the run's real signal (`run.abort.signal.addEventListener("abort", () => callAbort.abort(), {once:true})`) AND self-aborted the moment the internal idle timer fires (right before `settle(toolError(...))`); pass `callAbort.signal` as `ctx.signal` to `tool.execute()` instead of the bare run signal. Every existing server tool that already respects `ctx.signal` benefits identically (a tool told "you've timed out" via abort, not just a discarded return value) ‚Äî this is not a delegation-only carve-out.

**One consequence**: `runInnerLoop`'s own `signal` (used above) becomes `ctx.signal` ‚Äî now correctly scoped to "this call's own lifetime," not the whole run's ‚Äî which is why the earlier `signal` bullet points here instead of `parentRun.abort.signal`.

**Depth guard** (generalizes `MAX_DELEGATE_DEPTH`, `runner.ts:21`, today checked only inside `delegate_to_developer`):
- Add `delegationDepth?: number` to `ToolContext` (`tools.ts:19-26`) and to `AgentLoopDeps` (`agent-loop.ts:54-74`); `runServerTool`'s ctx construction (`agent-loop.ts:~266`) passes `delegationDepth: deps.delegationDepth ?? 0` through.
- `agent_delegate`/`dev_delegate` read `ctx.delegationDepth ?? 0`; if `>= MAX_DELEGATE_DEPTH` (moved to `inner-loop.ts`, still `2`), return the in-band error `"Delegation depth limit reached; cannot nest another delegation."` (FR-024(b)) ‚Äî never call `runInnerLoop`.
- Otherwise pass `delegationDepth: depth + 1` into the nested `runAgentLoop` invocation, so a delegate that itself calls `agent_delegate` sees the incremented depth ‚Äî applies uniformly to named/ephemeral/surface chains (Example 5), a real behavior change from today (currently only `delegate_to_developer` threads depth at all).

**Gate construction per delegation kind** (`src/lib/assistant/delegation-gate.ts`, new, or inlined in `subagents.ts`):
- **Named**: `gateFor(agentId)` ‚Äî unchanged, reused as-is (FR-005).
- **Ephemeral**: `{ allow: serverOnly(parentGate.allow, tools), deferred: new Set(), registryIds: parentGate.registryIds, descriptions: parentGate.descriptions }` where `serverOnly(allow, tools) = new Set([...allow].filter((id) => tools[id]?.execution === "server"))` (FR-003/FR-004).
- **Surface**: `{ allow: new Set(surfaceAgent.toolNames), deferred: new Set(), registryIds: parentGate.registryIds, descriptions: parentGate.descriptions }` ‚Äî trivial, because `visibleTools()` (`tools.ts:53-71`) already treats any name NOT in `registryIds` (i.e. most Tier-2 surface tools, which are never added to `CAPABILITIES`) as unconditionally visible once it's in the passed `tools` map; only a "real" registered capability like `a2ui_render` needs to actually be in `allow`, which it is (FR-007, FR-025).

**System-prompt composition per kind** (the `systemPrompt()` field of `InnerLoopSpec`):
- **Named**: reuse `composeInstructions(agentId)` (`instructions.ts:24`) **unchanged** ‚Äî this is a deliberate small upgrade over today's legacy behavior (today's `runLocal` passes bare `agent.systemPrompt` with no default-prompt/memory/skills-index/mcp-index at all; that was never a stated requirement, it's just what the old code happened to do). Since FR-005's whole point is "resolved identically regardless of invocation context," composing a named agent's prompt the same way whether it's primary or delegated is the natural extension, not scope creep ‚Äî called out here so it isn't a silent, undocumented behavior change.
- **Ephemeral**: `agent.systemPrompt` verbatim + the same "## Skills" / "## MCP servers" index blocks `composeInstructions` builds, filtered to the inherited ids (FR-016) ‚Äî factor that block-building logic out of `instructions.ts` into a small shared helper so it isn't duplicated. **Careful with unset vs. empty** (FR-016's clarified wording): "inherit in full" means inherit the parent's `skills`/`mcp` fields AS CONFIGURED ‚Äî an *unset* parent field means the ephemeral agent may use ANY skill/MCP server (today's unchanged unset-means-all semantics), so the index-building helper must treat `undefined` as "build the index from every skill/server," not "skip the section." Only a parent field that is explicitly `[]` means "build no index / nothing inherited." Gating this correctly against `filterAllowed`/`isAllowed` (`instructions.ts:43,53`, already unset-aware) rather than a fresh `.length > 0` check avoids re-introducing the exact unset-vs-empty bug FR-003 was written to kill for tools. **No** default-prompt prepending, **no** memory snapshot (FR-017; an ephemeral agent has no identity to have memory of).
- **Surface**: `agent.systemPrompt` (supplied by the registering app) verbatim, nothing appended (FR-017).

**Model override** (FR-015, genuinely new capability ‚Äî see Current-state audit):
- Extend `getProviderConfig(modelOverride?: string)` (`provider.ts:46`) to substitute `modelOverride` for the resolved `model` field when provided, everything else (provider/apiKey/baseUrl) unchanged ‚Äî deliberately NOT a per-agent provider switch, just a model-name override within the single configured provider.
- Extend `StreamTurn`'s options (`agent-loop.ts:36-45`) with an optional `model?: string`, threaded by `streamModelTurn` (`model-turn.ts:22`) into the new `getProviderConfig(opts.model)` call.
- `runInnerLoop` passes `model: namedAgent?.model` (undefined for ephemeral/surface, per FR-015) by wrapping `streamModelTurn` in a closure that injects it, rather than modifying `AgentLoopDeps`'s shape further.

### Phase 2 ‚Äî Rewire `agent_delegate`, add `dev_delegate`

- **`tools/server/subagents.ts`**: `agent_delegate`'s `execute()` (lines 73-116 today) keeps its input schema, ephemeral-spec construction, and `encodeNested()` wrapping, but replaces the body: resolve named/ephemeral spec ‚Üí build gate + composeSystem + model per Phase 1 ‚Üí depth-guard check ‚Üí `runInnerLoop(...)` ‚Üí map result to the same `"[agent ¬∑ type] N step(s)\n\n<output>" + encodeNested(...)` shape it produces today (UI compatibility ‚Äî the nested-card renderer doesn't change; this envelope is exactly what FR-019's "informational content ... verbatim, wrapper unaffected" wording was clarified to describe). `type: "claude"` keeps calling `runClaudeAgent` completely unchanged (FR-024(a)) ‚Äî this function only handles `type: "local"`.
- **`contentOnly`**: input field is preserved, unused by the `local` path (as today), still meaningful for `type: "claude"` (FR-020 ‚Äî explicitly not touched).
- **New `dev_delegate` tool** (`tools/server/dev-delegate.ts` or folded into `agent-admin.ts`): fixed-target wrapper ‚Äî always resolves `getAgent("developer")`, same depth guard, same `runInnerLoop`, replacing the closure-built tool in `makeDelegateTool` (`runner.ts:27-50`). Registered in `assistantTools()` (`registry.ts`) so Build Studio-style agents get it via their normal `tools:` allowlist instead of a hardcoded per-run factory. **Naming note (not a rename)**: `runner.ts`'s doc comments/function names call this "the `delegate_to_developer` tool," but the ACTUAL registered id today is `dev_delegate` (`subagents/tools.ts:218`'s `DELEGATE_TO_DEVELOPER = "dev_delegate"` constant), and every seeded skill/agent already calls it that (`seed/agents/build-studio/AGENT.md:13`, `data/skills/bos-app/SKILL.md`, `data/skills/feature-wizard/SKILL.md`, `capabilities-registry.ts:131`'s registered capability id). Keeping the name `dev_delegate` here is completing an already-established id, not introducing a breaking rename ‚Äî verified directly against the code before writing this, not assumed.
- **Tool-id validation surfacing** (FR-023): when an agent (named) is loaded (`getAgent`/`listSubAgents`, `subagents/store.ts`) or a gate is built (`gateFor`, `gate.ts:12`), any configured `tools`/`deferredTools` id not present in `assistantTools()` is logged (`logger().error(...)`) and collected into a per-agent "unresolved ids" list surfaced in Settings ‚Üí Agents (existing `AssistantTab`/agent editor ‚Äî exact widget left to implementation, not a new page). Valid ids continue to work (partial, not total, failure ‚Äî Edge Case).

If the default "assistant" personality should actually gain scheduler access as part of this migration, that is a deliberate seeded-allowlist change (see the migration-audit step below), not a side effect of the port.

This is real, concrete cleanup work this migration surfaces, not a hypothetical.

- `runSubAgent` (`runner.ts:157-179`) shrinks to just the `type === "claude"` branch (`runClaudeAgent`) ‚Äî `agent_delegate`/`dev_delegate` stop calling `runSubAgent` for `type: "local"` entirely, calling `runInnerLoop` directly instead. Keep the function name/shape for the claude path so `claude-runner.ts` and its callers don't need to change.
- Legacy VFS tools (`file_list/read/write/mkdir` server variants) and `web_fetch`/`web_search` duplication in `subagents/tools.ts` are deleted outright with no replacement ‚Äî v2 already has frontend VFS tools and `web-search.ts`'s `web_search`/`web_fetch`, used identically by any delegate via the shared registry.

### Phase 4 ‚Äî Surface agents

- **`src/lib/assistant/client/surface-agents.ts` (new)**, mirroring `client/surface-tools.ts` (`registerAppSurfaceTools`/`unregisterAppSurfaceTools`/`getActiveSurfaceToolDeclarations`/`findSurfaceToolHandler`) exactly:
  - `registerSurfaceAgent(windowId, {name, description, systemPrompt, toolNames})` ‚Üí derives `id = slugify(name)`. **Collision handling, corrected to match FR-023's actual (registration-time) requirement**: before accepting the registration, `fetch("/api/subagents")` (the existing endpoint, `GET` ‚Üí `{subAgents: Agent[]}`, already used by Settings ‚Äî no new server code needed for this check) and compare `id` against the returned persisted ids. A collision is REJECTED synchronously ‚Äî the registration call throws/returns an error the app's mount effect surfaces (console.error at minimum; the app MAY show its own toast), and the surface agent is NEVER added to the client-side registry at all. This is the check FR-023 actually asks for ("in-band rejection ... at registration time"), not a background log. A collision with a **currently-registered surface agent from a different window** (not persisted) is handled locally and non-fatally: append a short `windowId`-derived suffix (FR-022) and proceed.
  - `unregisterSurfaceAgent(windowId)`, `getActiveSurfaceAgents()`, `onSurfaceAgentsChanged` ‚Äî same change-notification pattern as `surface-tools.ts:34-44`.
- **`run-manager.ts`**: add a `Run`-scoped `agents: Map<string, SurfaceAgentEntry>` field (parallels `run.tools`) and a `toolTimeoutMs: number` field (Phase 1's linked-abort fix), plus `addSurfaceAgents(run, agents)` (parallels `addSurfaceTools`, `run-manager.ts:180-184`) ‚Äî additive-only, same "existing entries never overwritten" rule **deliberately mirroring surface *tools*' existing behavior**: verified directly (`run-manager.ts`'s `addSurfaceTools` never removes an entry either, and the existing `/api/assistant/runs/[runId]/surface-tools` route just re-merges whatever the client currently has), so an in-flight run not immediately losing a since-unregistered surface agent is accepted, pre-existing behavior for tools too ‚Äî not a new gap introduced for agents. This is why FR-010 was clarified to "any run started after that point," not "instantly, everywhere."
- **`start-run.ts`**: snapshot `getActiveSurfaceAgents()` at run start (parallels line 89's `manager.addSurfaceTools(run, opts.surfaceTools ?? [])`). A **second, backstop-only** persisted-id collision check happens here too (server-side, via `listSubAgents()`) ‚Äî this catches ONLY the race where a persisted agent was created AFTER a surface agent already passed its client-side check (FR-023's accepted exception; the client-side check above is the PRIMARY mechanism, not this one). A backstop collision is logged and the surface agent is dropped from that run's `run.agents`, rather than crashing run startup.
- **Mid-run live-push wiring (`FR-010`'s "live-pushed mid-run" ‚Äî the plan's original draft omitted this; adding it now)**: mirror the existing surface-*tools* push exactly.
  - New route `src/app/api/assistant/runs/[runId]/surface-agents/route.ts`: `POST { agents: SurfaceAgentPayload[] }` ‚Üí `runManager().get(runId)` ‚Üí `manager.addSurfaceAgents(run, agents)`, modeled 1:1 on the existing `.../surface-tools/route.ts`.
  - `client/run-client.ts`: add `pushSurfaceAgents(runId, agents)` + `flushSurfaceAgents(runId)` (mirroring `pushSurfaceTools`/`flushSurfaceTools`, `run-client.ts:34-40,56-66`) and subscribe to the new module's `onSurfaceAgentsChanged` the same way `onSurfaceToolsChanged` is already subscribed (`run-client.ts:71-73`), so a surface agent registered mid-run (e.g. right after `ui_preview_open`) becomes delegatable within that SAME run, not just the conversation's next one.
- **`discovery.ts`**: `find_agent` (`discovery.ts:43-59`) and `agent_list` (`subagents.ts:21-29`) both need the calling run's `run.agents` threaded in (today `discoveryTools()`/`subAgentTools()` don't receive the `Run` at all ‚Äî this is the one place a genuinely new parameter has to thread through `assistantTools()`'s call sites). Merge `listSubAgents()` (persisted) with `run.agents` (surface), tag each result `scope: "persisted" | "surface"` (FR-010), and score surface agents' `description` field the same way `scoreAgent()` already scores persisted ones.
- **`agent_delegate`**: resolution order checks the persisted roster (`getAgent`) **before** the run's surface-agent map (FR-022) ‚Äî a named agent id always wins if both somehow exist.
- **UI Preview** (`src/apps/ui-preview/`): first real registration, `{name: "Generative UI Agent", description: "...", systemPrompt: "...", toolNames: ["a2ui_render", "ui_preview_render"]}` (Example 2) ‚Äî `systemPrompt` is a variant of the app's existing generation-prompt context, not the full A2UI catalog description dump used for `a2ui_render`'s own internal sub-agent prompt ‚Äî keep those two prompts distinct.

### Logging integration (FR-027)

Consolidates every logging call site introduced across Phases 1‚Äì4 in one place, so implementation doesn't have to reconstruct "everywhere this spec touches observability" from scattered mentions. All server-side calls use `logger()` (`@/lib/logging`); the one client-side call uses `clog()` (`src/lib/logging/client/browser-logger.ts`) ‚Äî both already ship to the same central timeline (Current-state audit). New components, following the existing dotted `<area>.<subarea>` convention (`assistant.run`, `assistant.hooks`): **`assistant.delegate`**, **`assistant.surface-agents`**, **`assistant.tools`**.

- **Delegation lifecycle** (`inner-loop.ts` or `agent_delegate`/`dev_delegate`'s `execute()`, wrapping the `runInnerLoop` call): on start, `logger().log({level:"info", component:"assistant.delegate", conversation: ctx.conversationId, msg:"delegation started", data:{ kind: "named"|"ephemeral"|"surface", agentId, depth, maxSteps }})`. On finish, the same shape with `msg: `delegation finished: ${reason}``, `data: { agentId, kind, steps, reason }`, `level: reason==="error" ? "error" : "info"`, and (on error) `err: result.error` ‚Äî this is a direct mirror of `start-run.ts:98-104,130-136`'s existing "run started"/"run finished: reason" pattern, deliberately, for consistency in Settings ‚Üí Logs. **Do not log the `task` string or any tool arguments/results** ‚Äî metadata only, matching `assistant.run`'s existing precedent of never logging message content.
- **Depth-guard rejection** (`inner-loop.ts`, right before returning the in-band error, Phase 2/T017 in `tasks.md`): `logger().warn("assistant.delegate", "delegation depth limit reached", { conversationId, agentId, depth })`. Today this is 100% silent (the legacy `delegate_to_developer`/`dev_delegate` guard, `runner.ts:42`, has never logged anything) ‚Äî this is a real gap being closed, not decoration.
- **Named-agent tool-id validation** (FR-023, `gate.ts`'s `gateFor` or `subagents/store.ts`'s load path, Phase 2): `logger().warn("assistant.agents", "agent references unresolved tool ids", { agentId, unresolvedIds })`, in addition to the Settings-visible surfacing already speced.
- **Surface-agent collision** (FR-022/FR-023, Phase 4): client-side rejection in `client/surface-agents.ts`'s `registerSurfaceAgent` ‚Äî `clog("warn", "assistant.surface-agents", "surface agent registration rejected: id collides with a persisted agent", { windowId, id, name })` (ships centrally with zero new endpoint work ‚Äî confirmed in the Current-state audit). Server-side backstop in `start-run.ts` ‚Äî `logger().warn("assistant.surface-agents", "surface agent dropped at run start: persisted-id collision", { runId, conversationId, windowId, id })`.
- **Server-tool idle-timeout** (general ‚Äî every server tool, not delegation-specific; the linked-abort fix, Phase 1/T005 in `tasks.md`): right where `runServerTool`'s internal idle timer fires (`agent-loop.ts`, alongside `settle(toolError(...))` and the new `callAbort.abort()`): `logger().warn("assistant.tools", "server tool timed out", { conversationId, agentId, tool: tool.name, timeoutMs })`. This is currently invisible outside the in-band error string returned to the model ‚Äî for an unattended/headless delegation (nobody watching the chat), that in-band string alone leaves no operational trail at all.
- **What NOT to log**: individual tool calls/results, `step_started`/`text_delta` events, or anything already carried by the existing SSE event stream + persisted transcript ‚Äî central logging is for operational/debugging visibility across time (a developer querying Settings ‚Üí Logs later), not a duplicate of the conversation record. Logging every step of every delegation would also violate the "no full task/tool-argument content" rule above at high volume for no benefit.
- **Failure mode**: every call above follows the existing `logger()`/`clog()` contract ‚Äî fire-and-forget, never throws, a sink outage never affects delegation behavior (Current-state audit; 017's own requirement). No new try/catch is needed around these calls specifically, but they must never be placed where their (extremely unlikely) failure could block the in-band error/result path that already satisfies FR-020/FR-021.

## Risks / Open Items

1. **Model override (FR-015) is new, not a port** ‚Äî touches `provider.ts`/`model-turn.ts`, both used by the primary run path too. Low risk (purely additive optional parameter, default behavior unchanged when omitted) but worth its own focused test: a named agent with `model: "..."` set, delegated to, actually issues its API call with that model string.
2. **Named-agent prompt composition upgrade** (`composeInstructions` reused for delegated named agents, not just primary) is a deliberate, small behavior change beyond what FR-005 literally requires ‚Äî flagged in Phase 1's design notes rather than silently shipped, since a named agent delegated to today gets a bare prompt and will now additionally get memory/skills-index/mcp-index/default-prompt. If this is unwanted, the alternative (keep delegated named agents on bare `agent.systemPrompt`) is a one-line change in `delegation-gate.ts`.
3. **`runServerTool`'s linked-abort fix touches a shared file used by every server tool call, not just delegation** ‚Äî the per-call `AbortController` construction in `agent-loop.ts` (Phase 1) changes what `ctx.signal` IS for every existing server tool, not just `agent_delegate`/`dev_delegate`. Behaviorally this should be a strict improvement (any tool respecting `ctx.signal` now also gets told about its OWN timeout, not just run-level cancellation) ‚Äî but because it's a shared code path, it deserves its own focused test pass (a non-delegation server tool hitting its timeout still behaves identically) before folding it in with the delegation work, not just verified via delegation's own tests.
4. **`find_agent`/`agent_list`/`discoveryTools()` need the `Run` threaded in** ‚Äî today's `assistantTools()` call sites don't carry a `Run` reference into tool construction; Phase 4 is the one place a real new parameter has to flow through `registry.ts`'s composition, worth extra review since `assistantTools()` is a process-wide cache (`registry.ts:33` `let cache`) ‚Äî the `Run`-specific data (surface agents) must NOT be baked into that cache; it has to be read per-call from context (`ctx`), not from the cached tool map itself.
5. **Migration audit for stragglers**: before deleting `subagents/tools.ts`, grep for any other consumer beyond `dev-source.ts` (spec-tools' Build-Studio-only gating difference, mentioned in the spec's Assumptions, needs an explicit reconciliation decision ‚Äî v2 already gates spec tools to Build-Studio-style agents via normal allowlisting, which is arguably more correct than legacy's "hand it to any agent"; confirm this is intentional, not an oversight, before relying on it). Combine with the seeded-allowlist audit (Phase 3) ‚Äî same pass, same files.
6. **Surface-agent registration-time collision check depends on a network round-trip from a `"use client"` module** (`fetch("/api/subagents")` in `registerSurfaceAgent`) ‚Äî this makes registration asynchronous where `registerAppSurfaceTools` (the tools sibling it mirrors) is synchronous. Acceptable (app mount effects are already async-tolerant), but means there's a brief window between a window mounting and its surface agent becoming checked/registered; not a correctness problem (the agent simply isn't discoverable yet during that window) but worth noting so a test doesn't assume synchronous registration.
7. **New log volume is low but not zero** ‚Äî delegation start/finish is one pair of records per `agent_delegate`/`dev_delegate` call (not per inner step), so volume scales with delegation frequency, not delegation depth/length. No sampling/rate-limiting is proposed because the volume is bounded by the same thing that bounds tool calls generally (FR-021's strict sequentiality) ‚Äî if this ever proves noisy in practice, the fix is a log-level bump (`info`‚Üí`debug` for the start event) in Settings, not a new mechanism.
